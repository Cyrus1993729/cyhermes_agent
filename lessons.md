### L3 compaction 摘要注入用户消息
- **来源**: 2026-07-03 compaction 消息注入 bug 修复
- **规则**: 压缩摘要角色交替死锁时，不能把摘要合并到用户当前消息。应 prepend 到 head 末尾的 assistant 消息。注意两种死锁场景都需覆盖（head=assistant+tail=user 和 head=user+tail=assistant）
- **复现**: 1次（用户查进度，Agent 却启动下午的小红书分析任务）
- **状态**: active

### L8 L1审查API超时降级方案
- **来源**: 2026-07-13 黄金周报
- **规则**: qwen-bailian API超时时（直连+代理均失败），应在task-wrapup自检中执行人工对照契约逐条自检，作为降级方案。自检标注为"L1 API不可达，人工自检替代"
- **复现**: 1次
- **状态**: active

### L9 L1审查RUBRIC必须包含数值逻辑维度
- **来源**: 2026-07-13 黄金周报复盘
- **规则**: qwen_review.py 的 RUBRIC 必须明确要求审查数值逻辑，不可仅审查形式完成度。必须包含四子维度：a)符号方向(A<B→差为负) b)跨小节勾稽(同一组数多处一致性) c)口径一致性(同一概念全文统一术语) d)基本算术(数量级+勾稽关系)。缺失此维度时，即使千问推理能力强也不会主动做数值验证，导致符号错误/术语混用等低级错误遗漏
- **复现**: 1次（4.2 符号+0.14%应为-0.14%，6轮L1未发现，Opus第1轮即发现）
- **状态**: active

### L10 审查阶段应全程自动，不做伪阻断
- **来源**: 2026-07-13 黄金周报复盘
- **规则**: 审查阶段（L1/Opus）全程自动。每层最多3轮（L1 3轮+Opus 3轮=合计最多6轮）。第4轮启动前停下。CONDITIONAL算审查轮次，只有PASS不算。每层独立计数不跨层累计。停下条件：同一审查层连续3轮未PASS。根因：Agent长上下文执行惯性导致"假停顿真继续"
- **复现**: 2次（L1 第3轮后说停下但自修了；Opus 第1轮直接修没问）
- **状态**: active

### L11 Telegram 发送池耗尽需同时 drain polling + send 两个池
- **来源**: 2026-07-13 Telegram 投递故障诊断
- **规则**: 代理导致 httpx 连接 wedged 时，现有 `_drain_polling_connections()` 只清空 `_request[0]`（polling），`_request[1]`（send/edit）池满无人管。症状：入站正常，出站全堵（pool timeout 5 分钟）。修复：新增 `_drain_send_connections()`，在 pool timeout 检测到时异步清池
- **复现**: 1次（22:24-22:29 所有出站失败，重启恢复）
- **状态**: active

### L5 L1审查对齐——交付物分类必须与契约严格一致
- **来源**: 2026-07-04 Skill 微信→Telegram 迁移
- **规则**: 交付物中 D1/D2/D3 的分类数量必须与契约完全对应。甲类(8)+丙类(1)不能混成"9个skill"；额外展示的证据（如未改动的cron job）必须显式标注"未改动"

- **状态**: active

### L6 新增数值参数必须附推导依据
- **来源**: 2026-07-04 Skill 微信→Telegram 迁移
- **规则**: 在任何交付物中新增数值阈值/参数（如分段字数上限），必须附第一性原理推导（API上限→扣除格式化开销→净空间）和可靠来源（官方API文档）
- **复现**: 1次（L1 第1轮 CONDITIONAL — 3500字阈值缺少推导）
- **状态**: active

### L7 契约验收标准应写精确动作
- **来源**: 2026-07-04 文件治理规范
- **规则**: 写契约验收标准时，用精确动作而非抽象描述。如"step 6 执行 `rm -f workflow_*.md`"优于"创建前清孤儿";"hermes cron list 输出显示 deliver=origin"优于"已改为origin"。L1 审查只能验证可执行的事实，不能推断意图。
- **复现**: 1次（L1 FAIL — 清理孤儿动作不在指定 step）
- **状态**: active

### L12 高德静态地图 polyline 必须从 direction API 取沿路轨迹
- **来源**: 2026-07-17 WAIC 出行路线图
- **规则**: 用高德静态地图画驾车路线时，不能只用起点+终点坐标（画出来是直线）。必须调用 direction API `extensions=all` 取完整 polyline 坐标串，填入 staticmap `paths` 参数。跨城长路线需等距抽稀至≤100点（GET URL ~8000字符限制）。
- **复现**: 1次（Opus审查发现当前图是直线非沿路，用户无法用）
- **状态**: active

### L13 高德静态地图不支持虚线，需 PIL 叠加
- **来源**: 2026-07-17 WAIC 出行路线图
- **规则**: 高德 staticmap `paths` 参数无 dash 字段，所有线渲染为实线。如需虚线（如步行段标注），先让 staticmap 画实线作为底层，再用 PIL ImageDraw 在相同像素坐标叠画虚线。
- **复现**: 1次
- **状态**: active

### L14 L1 审查对交付物文本要求精确逐字匹配契约
- **来源**: 2026-07-17 WAIC 出行路线图
- **规则**: L1 审查会严格对照契约逐字比对交付物中的文字内容。底部栏路线描述、时间线文案即使多一个路名（如多了"乌盆港路"）或多一个修饰词（如"芯汇出发"vs"出发"）都会判 FAIL。如果想让交付物文字更详细，需先修改契约。
- **复现**: 1次（2轮 L1 FAIL 均因文字不完全匹配契约）
- **状态**: active
