# 微信交付三难困境

2026-07-03 实战发现，当报告超过单条微信消息容量时，以下三个需求**物理上无法同时满足**：

| 需求 | 状态 |
|:---|:---|
| 完整内容在微信正文里读完 | 用户要求 |
| 零额外操作（不输数字、不回"继续"、不点文件） | 用户要求 |
| Agent 控制发送节奏避免限流 | **做不到**——Agent 一次回复 = 网关一次投递 |

## 死结链条

```
内容长 → 必须拆多条 →
  要保证完整送达 → 需要控制节奏 →
    但 Agent 控制不了节奏 + 微信有限流 →
      死结
```

## 探索过的出路及结果

| 方案 | 结论 |
|:---|:---|
| 段间间隔 3-5 秒 | ❌ Agent 做不到（网关一次性投递） |
| 摘要 + 用户回复数字拉取段落（pull） | ❌ 用户不接受手动操作 |
| 压缩成决策自足的短稿 | ❌ 用户不接受信息量缩减 |
| MEDIA 文件交付 | ⚠️ 用户很少点开，仅作备份 |
| 维持原分段方案（分段发送，接受偶尔丢失） | ✅ 用户选择 |

## Opus 的最终评估

> "不是方案没想好，是约束把出口都封了。"

唯一可行的部分方案：
- 每条消息 = 一个自足逻辑单元（不跨消息拆同一论点）
- 关键结论放第 1 段（即使后面丢了，核心信息已到）
- 用户反馈丢失时主动重发

## 维持原方案的交付纪律

1. 自检摘要绝不发给用户（内部产物）
2. 一条回复只发一段，不塞额外内容
3. MEDIA 单独一条发
4. 用户反馈没收到 → 立即重发
