# Loop Engineering：从 PlanWeave 到 Hermes 的设计洞察

> 来源：2026-07-03 两轮 Opus 咨询 + 红队审查。核心议题——在不改 Hermes 源码的前提下，如何借鉴 PlanWeave 的设计思想。

---

## 根本洞察：文件中心 vs 对话中心

| | Hermes 当前（对话中心） | PlanWeave（文件中心） |
|:---|:---|:---|
| 真相在哪 | 聊天窗口里 | 磁盘文件上 |
| Agent 是什么 | 记忆体 | 临时读写器 |
| 会话结束 | 状态丢失 | 状态在文件里 |
| 审查 | 对话里的可选动作 | 任务图上的强制节点 |

所有痛点（cron 失忆、产物难找、审查漏、子代理踩文件）根因一致：真相活在对话窗口，不在文件系统。

---

## 核心设计原则

### 1. 追加式不可变记录（Append-only Journal）

**关键洞察**（红队问题 4）：靠 Agent 自律维护可变 state.json 是伪命题。Agent 不是 PlanWeave 的代码引擎，一定会忘写、写错、被打断。

**解法**：文件**只追加不改写**。状态靠读最后一行推导，不靠维护。

```
journal.ndjson: 每行 {"t":"时间","task":"任务","out":"产物路径","status":"ok|fail"}
cursors.json:   每次追加最新水位线，不回头改旧值
```

漏写一行 = 少一条记录，不会污染全局。绝不会出现「改了一半」的脏态。

### 2. 出处门（Provenance Gate）

替代关键词路由。不分类任务话题，只在交付前自检：「用户会不会拿这里的事实/数字做决策？」

- 会 → 必须附来源（URL 或「来源：xx，抓取于 xx 时」）
- 无法取得来源 → 标注「未核实」，降级为提示

好处：不依赖词表，不猜任务类型，每次都能做。

### 3. 一次性 cron run → 消除常驻循环

Opus 原方案有 Coordinator 循环（主 Agent 连续多轮：读状态→派工→等结果→审查→写状态）。红队指出：单 Agent 单对话窗口里循环物理上跑不了——用户随时插话，claimed 任务就悬着。

**修正**：Coordinator 换成 cron 一次性 run。每次拉起全新 Agent → 读 cursor → 干一件 → 追加一行日志 → 退出。没有循环，就没有被打断的循环。

### 4. 黄红同格式 → 不丢活

原方案中黄道和红道产物格式不同，棘轮升红会导致前面干的活白干。修正：所有任务产出同一种格式（带出处 + 流水账附加行）。「升红」退化为两个零成本动作——触发出处门 + 交付前暂停确认。

## 砍掉的方案（及原因）

| 被砍 | 原因 |
|:---|:---|
| 三车道路由（route.py） | 关键词太脆，漏掉的比命中的多；应改为出处门 |
| 打分表 | 过度设计，一个Agent维护不了 |
| Coordinator 循环 | 单对话跑不了 |
| manifest.json + state.json | Agent 自律不可靠，改为 append-only |
| 棘轮升红 | 黄红格式不同导致白干；统一格式后无需棘轮 |
| 修理工 skill | 复杂度过高，先跑通基本流程 |
