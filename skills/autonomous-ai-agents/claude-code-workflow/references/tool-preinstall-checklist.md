# 工具/插件安装前评估清单

## 触发条件

当考虑安装新工具、npm 包、Skill、或对系统做配置变更时，执行以下 4 步检查，**不要直接安装**。

## 4 步评估法

### 1. 备份现有状态

- 确认会受影响的文件/目录
- 完整备份到桌面或其他安全位置
- 记录备份路径，以便回滚

```bash
# 示例：备份 skills 目录
cp -r ~/.hermes/skills ~/Desktop/skills-backup-$(date +%Y%m%d_%H%M%S)
```

### 2. 审查安装脚本/源码

**不要信任 "一键安装" 等营销话术。** 必须查看安装器实际做了什么：

- npm 包：用 `npm pack` + `tar xzf` 下载并解压
- 重点看：`bin/` 目录下的安装脚本、`package.json` 的 scripts
- 确认：文件写入路径、是否会覆盖已有文件、是否有 sentinel 标记
- Shell 脚本：逐行阅读，确认每个 `cp`/`rm`/`write` 的目标路径

```bash
# npm 包源码审查
mkdir inspect && cd inspect
npm pack <package-name>
tar xzf *.tgz
# 然后阅读 package/bin/install.js 等关键文件
```

### 3. 确认卸载/回滚方式

- 有没有 `--uninstall` 选项？有没有 `--dry-run`？
- 安装器是否用哨兵标记（如 `<!-- begin -->...<!-- end -->`）隔离追加内容？
- 如果没有干净的卸载方式，手动回滚是否可行？

### 4. 隔离测试核心能力

**不要基于 README 描述做假设。** 对宣传中最吸引你的功能，先隔离验证：

- 检查是否有隐藏的外部依赖（如 `workflow-runner` 依赖 `agency-agents-zh`）
- 检查功能是否匹配你的实际使用场景（如项目级安装 vs 全局安装）
- 在非生产环境先试跑

## 决策框架

| 检查结果 | 行动 |
|:---|:---|
| 全部通过 | 可以安装，但先汇报用户确认 |
| 有不确定项 | 向用户报告风险，让用户决策 |
| 有明确不适合的地方 | 建议不装，解释原因 |

## 实战案例：superpowers-zh

**宣传：** "一条命令安装，20个中文 Skill"

**4 步检查结果：**
1. ✅ 备份 636 个文件到桌面
2. ⚠️ 安装是项目级（`{cwd}/.hermes/skills/`），用户需要全局安装
3. ✅ 有 `--uninstall`，哨兵标记精确切除
4. ❌ workflow-runner 需要外部依赖 `agency-agents-zh`，不独立可用

**最终决定：** 不装。价值和成本不匹配用户实际场景。
