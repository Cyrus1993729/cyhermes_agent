# L2 独立评审模式（Claude Sonnet 做技术评审）

## 适用场景

当面临以下决策时，在动手之前先让 Claude Sonnet 做一次独立评审：
- 是否安装/采用某个新工具或框架
- 对某个技术方案或建议进行第二意见验证
- 用户对某项决策不确定，希望有外部视角

## 为什么用 Sonnet 而不是 Opus？

- **L2 级任务**（已有框架的验证）用 Sonnet 足够，速度快、成本低
- **L3 级任务**（新领域 + 多维复杂 + 框架可复用 + 高利害）才用 Opus
- 原则：评审已有分析用 Sonnet，从零设计框架用 Opus

## 完整模板

### Step 1: 构建给 Claude 的问题

```
你是被请来做技术评审的 Claude。请对以下判断给出你的独立意见
（同意/不同意/部分同意，并解释理由）：

== 背景 ==
[描述用户的使用场景、技术水平、当前工具链]

== 项目 ==
[项目名称、仓库、核心功能、关键数据]

== 我的判断 ==
我认为 [你的结论]。理由：
1. [理由1]
2. [理由2]
...

== 担忧 ==
1. [担忧1]
2. [担忧2]
...

== 请给出你的独立判断 ==
1. 是否值得 [安装/采用]？
2. 我的分析有没有遗漏或偏差？
3. [行动] 之前需要注意什么？

请用中文回复，直接、坦率。不确定的地方就说"不确定"。
```

### Step 2: 执行

```bash
export HTTP_PROXY="http://127.0.0.1:7897" HTTPS_PROXY="http://127.0.0.1:7897"
claude -p '[上述问题]' --model sonnet --max-turns 5 --output-format text
```

### Step 3: 对比分析

拿到 Claude 的评审后，逐条对比：
- Claude 同意的 → 信心增强
- Claude 不同意且有理由的 → 审视自己的分析是否遗漏
- Claude 指出你没想到的 → 高价值收获

### Step 4: 汇报给用户

向用户汇报时应包含：
1. Claude 的总体判断（同意/部分同意/不同意）
2. Claude 指出的关键问题
3. 你的修正后建议
4. 明确下一步行动

## 实战案例：superpowers-zh 评审

**场景：** 用户考虑安装 superpowers-zh，我先做了分析认为值得装。

**Claude Sonnet 的评审结果：** 部分同意。指出了我的分析漏洞：
- "一键安装零配置"是营销话术，实际安装行为不透明
- Skill 冲突风险我低估了
- workflow-runner 的实际能力未验证

**修正后行动：** 先做4步检查（备份→源码审查→卸载验证→能力测试），发现 workflow-runner 需要外部依赖、项目级安装不匹配用户全局设置，最终决定不装。

**关键教训：** 如果没有 Claude 的独立评审，可能就直接装了，之后才发现问题。

## 边界

- 这个模式用于**评审已有分析**，不是「让 Claude 代替你做分析」
- Claude 的角色是「发现你漏了什么」，不是「替你做决定」
- 最终决策权在用户，你在中间做综合判断
