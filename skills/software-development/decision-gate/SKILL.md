# Decision Gate（决策闸门）

## 触发条件

本规则适用于所有与用户交互的场景。当以下情况发生时，必须触发决策闸门：

- 研究/搜索后发现了可执行的操作（安装工具、修改配置、创建文件等）
- 找到了解决方案，准备动手实施
- 用户说"进展呢""怎么样了"等进度询问——**这是要汇报，不是授权**

## 规则（强制）

```
研究/发现 → 【闸门：向用户汇报发现了什么】 → 用户明确同意 → 动手
```

**闸门前：** 只能做只读操作（搜索、阅读、分析、理解）。  
**闸门后：** 用户说"装""做""试试""可以"等明确同意词后，才能执行写操作。

## 闸门汇报模板

向用户汇报发现时，包含：

1. **找到了什么**（项目名、仓库、关键数据）
2. **这东西是什么**（一句话定位）
3. **要做什么操作**（具体命令/改动）
4. **风险/副作用**（如有）
5. **明确询问**：「要不要我帮你装/做？」

## 禁止行为

- ❌ 搜索到项目后直接 `pip install` / `git clone` / 修改文件
- ❌ 把用户说"进展呢"当成"直接动手"
- ❌ 在汇报发现的同时偷偷执行操作
- ❌ 跳过汇报步骤，默认用户"肯定想要"

### ⚠️ 委托分析结果也走闸门（2026.7.1 新增）

**问题：** 用 `delegate_task` 委托另一个模型（如 Claude Opus）分析问题后，子agent 返回了完整的解决方案。因为方案看起来"已经由另一个模型给出"，容易产生「方案已经成熟，可以直接执行」的错觉，跳过用户确认。

**规则：委托分析结果 = 研究发现，同受闸门约束。**
- delegate_task 返回的方案 → 必须先展示给用户 → 等用户说"可以"/"做"/"试试" → 再执行
- 不能在 delegate_task 返回后紧接着就执行方案中的步骤
- 即使用户说过"让 Opus 给方案"，也不代表授权直接执行 Opus 的方案

### ⚠️ 非技术用户的技术决策 → 委托外部专家（2026.7.3 新增）

**问题：** 决策闸门揭示了多个技术方案（如 bug 修复的 A/B/C 三选一），但用户说"我是技术小白，不知道选哪个"。

**规则：用户不能评估方案 ≠ 跳过决策闸门。** 正确的路径是：

1. 用户明确说"你帮我问问 Opus/Claude" → 走外部专家咨询
2. 外部专家返回推荐 → 再次触发闸门，展示专家意见 + Agent 本地评审
3. 用户确认 → 执行

**反例（正确执行的案例，2026.7.3）：** compaction 注入 bug 的三个修复方案 → 用户说"我是技术小白，让 Opus 评估" → Opus 推荐了第四方案 → Agent 本地评审优化（prepend 替代 append）→ 用户确认 → 执行。链完整：没有因用户不懂就跳过闸门，没有因 Opus 说了就直接执行。

**注意：外部专家的意见仍需过闸门。** 不能把"Opus 说用方案 X"当成"用户授权执行方案 X"。

### ⚠️ 诊断完成 → 执行阶段的切换点（2026.7.3 新增）

**问题：** 涌现型任务天然没有"一开始就定义交付物"的阶段。诊断阶段是只读的、探索性的，但一旦找到根因和方案，就进入了执行阶段——这个切换点最容易跳过闸门。

**规则：任何外部模型给出的修复方案，Agent 必须先展示方案概要 + 契约 → 等用户确认 → 再动手。**

**不能在用户说"改吧"时直接跳到执行。** "改吧"不等于"我确认了方案的完整性和边界"。

### 🔀 涌现型任务的完整流程（2026.7.3）

```
┌─ 诊断阶段（免契约）──────────────┐
│ 搜索、读日志、看代码、问外部模型    │
│ 规则：只读、只分析、不动手          │
│ 终点：找到了根因 + 可执行的修复方案  │
└────────────┬────────────────────┘
             │ ← 用户说 "走流程" 触发切换
             ▼
┌─ 执行阶段（走完整闭环）────────────┐
│ ① sprint-contract（出契约）        │
│ ② decision-gate（用户确认方案）     │
│ ③ 执行                             │
│ ④ task-wrapup（审查+存档+交付）     │
│ ⑤ post-task-review（复盘）         │
└────────────────────────────────────┘
```

**触发词约定：**
- 用户说 **"走流程"** → 从诊断阶段切换到执行阶段，出 sprint-contract
- 用户说 **"收尾"** → 触发 task-wrapup + post-task-review

### ⚠️ 严格遵循用户指令顺序

用户说"A then B"，必须严格按顺序执行。不能颠倒、不能并行、不能跳过其中一步。

## ⚠️ 闸门与不稳定通道的冲突（2026.6.21）

**问题：** 通过微信（iLink）执行 `hermes gateway stop` 类操作时，「确认→执行」的 round-trip 可能被网关中断吞掉。

**处理规则：** 涉及 `gateway stop` 或会中断当前通信通道的操作：**不等确认**，用户表达意图后直接执行。

## 反例

用户要求研究 AgentReach → 搜到了项目 → 用户问"进展呢" → **直接 pip install 了** → 用户喊停。

正确做法：先汇报发现 → 等确认 → 再执行。
