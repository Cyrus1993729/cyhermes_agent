# Decision Gate（决策闸门）

## 触发条件

本规则适用于所有与用户交互的场景。当以下情况发生时，必须触发决策闸门：

- 研究/搜索后发现了可执行的操作（安装工具、修改配置、创建文件等）
- 找到了解决方案，准备动手实施
- 用户说"进展呢""怎么样了"等进度询问——**这是要汇报，不是授权**

## 规则（强制）

```
研究/发现 → 【闸门：向用户汇报发现了什么】 → 用户明确同意 → 动手
```

**闸门前：** 只能做只读操作（搜索、阅读、分析、理解）。  
**闸门后：** 用户说"装""做""试试""可以"等明确同意词后，才能执行写操作。

## 闸门汇报模板

向用户汇报发现时，包含：

1. **找到了什么**（项目名、仓库、关键数据）
2. **这东西是什么**（一句话定位）
3. **要做什么操作**（具体命令/改动）
4. **风险/副作用**（如有）
5. **明确询问**：「要不要我帮你装/做？」

## 禁止行为

- ❌ 搜索到项目后直接 `pip install` / `git clone` / 修改文件
- ❌ 把用户说"进展呢"当成"直接动手"
- ❌ 在汇报发现的同时偷偷执行操作
- ❌ 跳过汇报步骤，默认用户"肯定想要"

### ⚠️ 委托分析结果也走闸门（2026.7.1 新增）

**问题：** 用 `delegate_task` 委托另一个模型（如 Claude Opus）分析问题后，子agent 返回了完整的解决方案。因为方案看起来"已经由另一个模型给出"，容易产生「方案已经成熟，可以直接执行」的错觉，跳过用户确认。

**规则：委托分析结果 = 研究发现，同受闸门约束。**
- delegate_task 返回的方案 → 必须先展示给用户 → 等用户说"可以"/"做"/"试试" → 再执行
- 不能在 delegate_task 返回后紧接着就执行方案中的步骤
- 即使用户说过"让 Opus 给方案"，也不代表授权直接执行 Opus 的方案

**反例（本次会话）：** Opus 返回了 3 个策略方案 → 没有展示直接运行了 `xurl auth oauth2 --app hermes-agent`（Strategy A）→ 超时失败 → 用户指出"你都没有把方案发给我让我确认"。

### ⚠️ 严格遵循用户指令顺序

用户说"A then B"（如"先把模型切到v4pro，然后把问题发给opus"），必须严格按顺序执行。不能颠倒、不能并行、不能跳过其中一步直接做第二步。更不能在用户指定了目标模型后才用其他模型委派。

## ⚠️ 闸门与不稳定通道的冲突（2026.6.21 新发现）

**问题：** 通过微信（iLink）执行 `hermes gateway stop` 类操作时，「确认→执行」的 round-trip 可能被网关中断吞掉。本会话连续 3 次：用户确认执行 → 网关重启 → 执行窗口丢失 → 用户手动恢复连接。

**根因：**
1. iLink 频繁限流（本会话 429 次 rate limit），导致多段消息部分丢失
2. v0.14.0 网关不稳定，同一天 7 次 restart-interrupted session
3. 「停止微信通道」这个操作通过微信通道执行，存在逻辑悖论

**处理规则：**
- 涉及 `gateway stop` 或会中断当前通信通道的操作：**不等确认**，用户表达意图后直接执行。宁可承担少一次确认的风险，不让执行被网关中断吞掉。
- 或者：将操作写成 `.bat` 脚本放到桌面，让用户手动双击执行——绕过微信通道完全避免问题。详见 [`references/batch-script-pattern.md`](references/batch-script-pattern.md)
- 不涉及通信中断的操作：保持闸门规则不变

用户要求研究 AgentReach → 搜到了 Panniantong/Agent-Reach → 用户问"进展呢" → **我直接 pip install 了，没有先汇报搜索结果** → 用户喊停。

## 与 memory 中规则的关系

用户 profile 中已有：「流程：出方案→用户确认→动手」。本 skill 是该规则的**程序化执行**，确保 AI 在每一轮对话中都遵守，而非仅在记忆中存在。
