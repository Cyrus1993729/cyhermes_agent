# 设计审查工作流

## 什么时候用

用户在投入大规模工程之前，想用外部独立视角审视一个设计方案。核心原则：

> **不要让设计者和审查者用同一个模型。** 审查者必须独立、冷峻、不找补。

## 标准流程

### 1. 确认审查对象

用户通常有一份设计文档。确认：
- 文档路径
- 审查视角（红队/可实现性/安全/架构）
- 特殊约束

### 2. 选择审查模型

| 场景 | 推荐审查者 |
|------|-----------|
| 系统架构可实现性 | Claude Code（推理强，批判性好） |
| 安全/漏洞审查 | Claude Code |
| 多角度权衡 | 多个 subagent 并行，各自不同立场 |
| 快速轻量审查 | delegate_task subagent |

⚠️ 不要用当前会话的模型做审查——它会和设计者共享推理偏差。

### 3. 构造红队 Prompt

审查 prompt 的核心要素：

```text
1. 明确角色：红队审查员，不是顾问
2. 禁止行为：不找优点，不找补，不给安慰性结论
3. 审查维度（根据场景选择）：
   - 哪些模块根本实现不了？为什么？
   - 哪些复杂度被严重低估？
   - 最脆弱的环节在哪？崩了会怎样？
   - 有没有死循环/不可达路径？
   - 路由/判定逻辑是否清晰可判定？
   - 在当前框架（如 Hermes）实现有什么根本矛盾？
4. 严重度评级：致命/严重/中等/轻微
5. 总体判断：该不该投入工程？要不该，替代方案是什么？
6. 输出语言：中文（本用户偏好）
```

### 4. 传递设计文档

方式：
- 如果审查者能读文件：传路径
- 如果审查者不能读文件：传摘要 + 关键段落
- 注意 v3.2.1 的 Context Packet 模式（见 references/context-packet.md）

### 5. 解读结果

审查结果回来后的处理：
- 不要急于反驳或解释
- 先完整呈现给用户
- 区分「工程约束」（当前无法改变）vs「设计缺陷」（可以修复）
- 让用户判断哪些接受、哪些要深化讨论

## 常见陷阱

### 陷阱 1：用同一个模型审查自己的输出
LLM 审查 LLM 输出只能检查格式合规，不能验证内容正确性。必须换模型或换会话。

### 陷阱 2：审查 prompt 太客气
"请评价这个设计" 会得到平衡的、有赞誉有建议的回复。用 "你是红队，只找问题" 才能得到真实批判。

### 陷阱 3：忽略基础设施差距
设计方案中引用的工具/API 可能在现实中不存在（幽灵 API）。审查时必须逐一验证每个依赖是否真实可用。

### 陷阱 4：把方法论当成工程规格
精心撰写的流程文档 ≠ 可实现的系统设计。审查时要区分 "概念上自洽" 和 "工程上可行"。

## 典型案例

用户提交了一份研究院 Agent 设计文档（v3.1-DRE），包含六层架构、A/B/C 路由、委托式研究工程等。用 Claude Code 做红队审查，发现了 3 个致命问题（幽灵 API、无界返工循环、有状态 vs 无状态矛盾）和 4 个严重问题，结论是不该在当前阶段投入工程实现。

详见 `references/research-institute-methodology.md`。
