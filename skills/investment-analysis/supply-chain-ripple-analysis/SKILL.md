# 供应链涟漪分析

## 触发条件

用户要求分析某个公司/事件对产业链的影响，或说"像 Serenity 那样找出受益公司"、"分析 XX 的产业链上下游"、"XX 事件哪些上市公司会受益"。

## 核心理念（来自 Claude Opus 设计）

> **一个事件有涟漪，当且仅当它在产业链里"插入了一个新节点"，而这个节点别人必须围着它建东西。**

供应链本身就是一台"时间机器"——上游必须先于下游承诺产能。一波需求会沿供应链逆流留下一串时间错开的公开痕迹。

## 涟漪事件分级

| 级别 | 事件类型 | 涟漪强度 |
|------|---------|:---:|
| **S** | 私有平台公司大额融资、新工艺平台(PDK)发布、流片、标准联盟成立 | 最强 |
| **A** | 技术合作/联合开发、**代工厂签约**、客户产能预定/预付款 | 强 |
| **B** | 新品发布、路线图更新、展会 demo | 中等 |
| **C** | 财报数字、普通 PR、获奖 | 几乎无 |

**只盯 S 和 A。** 平台/代工/标准三类权重最高，因为它们天然是"被围绕"的节点。

## 三问过滤（判断是否启动全流程）

对任何新闻依次问：

1. **它是平台还是产品？** 别人要在它之上建东西→留下；纯终端→丢弃
2. **它的 BOM 深不深？** 有卡脖子层（资质门槛）→留下；纯组装→丢弃
3. **它有没有创造"新合格供应商坑位"？** 新平台=必须有人测它、封它、供它器件

三个都 yes = 产业链里程碑，启动全流程。

## 分析流程

### 阶段 1：事件检测与品类识别

用 Tavily 搜索事件细节，让 Claude 判断事件级别和品类。

```
Tavily: topic="news", days=30, search_depth="advanced", max_results=20
Query: "private company foundry partnership semiconductor 2026"
       "Series B OR Series C funding photonics OR silicon 2026"
       "new PDK OR platform tape-out 2026"
```

### 阶段 2：动态 BOM 生成（核心创新）

**不要预制 BOM 字典**——每次分析用 Claude 动态生成该品类的标准 BOM 架构。

Prompt 模板：
```
请为 [品类名称] 生成标准 BOM 展开：
- 列出每一层（设计→代工→核心器件→封装→测试→终端）
- 标注每层的卡脖子程度（高/中/低）和原因
- 标注每层的关键技术关键词（用于后续搜索）
```

### 阶段 3：逐层填坑（Tavily 搜索）

对 BOM 每一层，跑两轮搜索：

**3a. 直接关联搜索**（找有官宣关系的公司）
```
Query: ""<公司名>" <层关键词> partner OR supplier OR collaboration"
include_domains: sec.gov, prnewswire.com, businesswire.com, <公司IR域名>
```

**3b. 全层玩家搜索**（找该层所有上市公司，即使无官宣）
```
Query: "<品类> <层关键词> public listed company suppliers"
max_results: 20
```

### 阶段 4：证据定级

| 标签 | 含义 | 触发条件 |
|------|------|---------|
| **强（官方）** | 双方官宣/财报/SEC 点名 | 找到一手公告 |
| **中（交叉）** | 高管互相引用但无正式协议 | 新闻稿引言、行业报道 |
| **弱（推断）** | BOM 必要性推出，无任何公开关联 | 仅靠参考架构 |
| **待验证** | 重要但还没查到 | 需要下一步动作 |

**输出时永远标明证据级别**——这比 Serenity 一条推文更专业。

### 阶段 5：排序输出

按"卡脖子程度 × 证据强度"排序，先排瓶颈层再排普通层。

## 领先信号类型（当需要更早期发现时）

Opus 识别的高价值公开信号（按领先时间排列）：

| 信号 | 领先 | 特点 |
|------|:---:|------|
| 技术论文/标准组织活动 | 1-3年 | 最早但最脏 |
| 招聘信息（特定工艺/设备岗位） | 6-18月 | **最被低估，结构化可抓取** |
| 专利申请与受让/引用 | 1-3年 | 需要专业知识 |
| 环评/用地/项目备案 | 1-2年 | 不可逆资本承诺 |
| 设备厂订单簿与交期 | 9-18月 | 下游产能硬预订 |
| 海关/提单数据 | 1-2季 | 实物在走货 |
| SEC Form D / 私有公司融资 | 6-18月 | **最接近 Serenity 的入手点** |
| 展会议程和 demo 排期 | 1-2月 | 合作已发生但未官宣 |

**关键认知**：清晰度和领先时间反向。最早信号最不可信。真正的 edge 是"多个早信号交叉确认"——招聘+备案+设备订单同时指向同一处，三个脏信号收敛成一个硬判断。

## 阶段 6：交叉确认（对"弱"标签公司升级验证）

对产业链推断的候选公司，用三个正交问题确认信号真实性：

| 要回答的问题 | 查什么 | 判断标准 |
|-------------|------|---------|
| 是不是真投钱了？ | 设备招标/订单（hybrid bonder、TCB、贴片机） | 钱已承诺=真 |
| 是不是量产级（非实验室）？ | 环评/备案里的产能数字（片/月、万只/年）、洁净室等级、资本开支 | 有产能数字+排污量=量产 |
| 是不是被真实客户拉动？ | 下游客户同期动作（合作公告、同类岗位招聘、路线图） | 有下游同步=需求真 |

**收敛规则**：
- 招聘 + 设备招标 + 带产能数字的备案 + 下游路线图 = 高置信度真实扩产
- 只有招聘 + 研报炒作，查不到设备/备案 = 画饼/噪音
- 招聘 + 设备（真投钱），但下游零动作 = 真投钱但需求存疑

**最锋利的反噪声检验：时间序列自洽性。** 真实扩产的信号有固定先后：环评/备案（最早，动土前）→ 设备招标（早 6-12月）→ 工程师招聘（产线建设/投产前）→ 海关到货 → 客户验厂认证 → 量产/operator 招聘。如果招聘出现了但前序的备案和设备订单查不到，基本是画饼。**信号出现的时间顺序错乱，本身就是 noise 的 tell。**

## Pitfalls

- **不要预制 BOM 字典**——每次用 Claude 动态生成，避免锁死分析领域
- **输出永远带证据级别标签**——不要对无证据的推断标注为"事实"
- **区分"官宣合作"和"产业链推断"**——两者都可以列出但必须标注清楚
- **不要用招聘 JD 做雷达/触发器**——假阳性太高，分不清替换性招聘（旧人走了）、研发性招聘（实验室级别）、画饼性招聘（没拿到钱就招人造势）。待定更好的雷达信号
- **Bing 搜索质量不如 Google/Tavily**——Tavily 返回了比手动 Bing 搜索多 3 个关键发现（Jabil 官宣、IQE 供应商、NewPhotonics 客户），优先用 Tavily
- **不要在"绕过 Google CAPTCHA"上花时间**——代理 IP 数据中心被标记，无解，用 Tavily API 代替
- **alpha 和抓取难度成正比**——最好抓的最没 edge（SEC API、股价数据人人有），最难抓的最有 alpha（环评/备案爬虫——分散在省市县三级、格式不统一——正因为难抓所以没人系统化看）。把工程预算砸在"难抓但强制留痕"的环评/备案爬虫上
- **Marvell/Sivers 案例**：Serenity 推文中 2/7 家公司零证据纯推断，不是所有他列出的都有真实合作关系

## Claude Opus 多轮咨询模式

当需要对方法论做框架级设计时，使用多轮分步咨询（不要一锅炖）：

- 第 1 轮：开放提问，让 Opus 自由思考（不预设框架）
- 第 2 轮：聚焦信号采集和交叉验证策略
- 第 3 轮及以后：逐层深入具体落地细节

每轮只聚焦一个子问题，用户确认后再发下一轮。
