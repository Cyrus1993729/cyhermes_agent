## 🎯 任务目标

## 🔄 实际流程
1. **Qwen OAuth 尝试**（失败）→ Qwen Code CLI v0.19.2 已删除 auth 命令
2. **切回 custom provider 方案 B** → 用阿里百炼 OpenAI 兼容端点替换 Anthropic 端点
3. **Opus 5 次重试**排查 401 → `--bare` 丢登录态、pipe 传参加上 YAML 被 bash 解析、max turns 耗尽 → 最终纯文本 prompt + `--allowedTools Read` 成功
4. **Opus 找到根因**：`anthropic_adapter.py` 对非 Anthropic 端点一律发 x-api-key，百炼要 Bearer
5. **方案 B 修配置** → 端点从 `/apps/anthropic` 切到 `/compatible-mode/v1`，`api_mode` 从 `anthropic_messages` 改为 `chat_completions`
6. **第二个 bug 发现**：MoA 配置格式 `provider: custom, model: qwen-bailian:qwen3.7-max` 错误 → `resolve_provider_client` 只在 Named Custom Providers 段查找 provider 字段，不解析 model 里的冒号前缀 → 改为 `provider: qwen-bailian, model: qwen3.7-max`
7. **MoA 测试通过**：三个模型 `/moa 回复OK` 全部正常
8. **黄金分析**：MoA 产出（DeepSeek 汇总 Qwen+Kimi 参考）vs Opus 4.8 独立分析 → 对比打分

## ⚠️ 踩过的坑
| 坑 | 后果 | 解决 |
|:---|:---|:---|
| `provider: custom + model: qwen-bailian:qwen3.7-max` 格式 | 冒号前缀被忽略，api_key 永远 None | 改为 `provider: qwen-bailian + model: qwen3.7-max` |
| Opus 第 1 次 prompt 含 YAML | bash 把缩进当命令 | 写文件再 pipe |
| Opus 第 2-3 次 max turns | `--bare` 无登录态 / 工作区太大 | 去掉 bare，加 `--allowedTools Read`，纯文本 prompt |
| Qwen OAuth 路径 | Qwen CLI 新版本删了 auth | 放弃，走 API key |
